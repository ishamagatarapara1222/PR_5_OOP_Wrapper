## 🧠 Project Description

**OOP Wrapper** is an advanced, enterprise-grade console application designed to simulate a real-world Employee Management Architecture using pure Object-Oriented Programming (OOP) paradigms. The primary focus of this system is to encapsulate sensitive business entities and manage structured data relationships through complex class blueprints, entirely discarding functional procedures.

The core application operates through an interactive menu-driven interface that enables administrators to provision, query, track, and manage heterogeneous human resource components (`Person`, `Employee`, `Manager`, and `Developer`). It serves as an ultimate showcase of structural object modeling, memory space allocation tracking, and runtime type analysis.

By wrapping low-level system properties inside secure getters/setters and executing polymorphically overridden behaviors across tiered inheritance structures, this utility delivers true software engineering design patterns. It provides a solid architectural blueprint for implementing enterprise access controls, object relational mappings, and secure micro-service logic.

## ⚙️ Core OOP Pillars Implemented

### 🔹 1. Encapsulation (Secure Data Hiding)
Attributes like Employee ID and Salary are securely locked as strictly private variables using Python's double-underscore (`__`) prefix. Access is heavily regulated and granted exclusively via validated **Getter & Setter** functions.

### 🔹 2. Multi-Level Inheritance
Establishes a neat hierarchical blueprint where child classes automatically inherit basic details from the master `Person` class, while appending specialized enterprise-level attributes based on specific corporate roles.

### 🔹 3. Polymorphism & Method Overriding
The `.display()` method is systematically overloaded and overridden across the entire subclass stack. At runtime, the system automatically triggers the specific subclass method to cleanly format properties without code duplication.

### 🔹 4. Overloading & Destructor Handling
Handles multiple ways of instantiation via custom parameter checks inside class constructors, mimicking constructor overloading. Additionally, explicit memory cleanup is validated using native `__del__` destructor workflows.

## 🧪 Execution Flow

1. Administrative engine boots up and initializes separate lists for entities.
2. Control interface provides option vectors for profile additions, role queries, or class validation.
3. System triggers custom constructors, handling default or loaded input parameters gracefully.
4. Profiles are retrieved and systematically evaluated via polymorphic display calls.
5. System-wide structural hierarchy is validated using native `issubclass()` assertions.
6. Memory cleanup hooks execute explicitly upon termination, safely wiping active collections.
